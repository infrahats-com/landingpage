# infrahats SEO Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make infrahats.com fully indexable, rich-result-eligible, and share-ready on every platform — **LinkedIn first** (that's where a DevOps consultancy's buyers are), plus Facebook/WhatsApp/Messenger, Slack, Discord, Telegram, iMessage, and Twitter/X — and get it registered with search engines at launch.

**Architecture:** All on-page SEO lives in `src/layouts/Layout.astro`, with head tags generated by the `astro-seo` package (one declarative `<SEO>` component instead of a dozen hand-maintained meta tags) plus hand-rolled JSON-LD. Platform coverage works like this: **LinkedIn, Facebook, WhatsApp, Messenger, Slack, Discord, Telegram, and iMessage all read Open Graph tags** — LinkedIn ignores `twitter:*` entirely — while Twitter/X reads its own card tags and falls back to OG. A complete OG set + one Twitter card therefore covers every platform. Static assets (OG image, touch icon) live in `public/`. Verification is grep-based against the built `dist/index.html`; off-page steps (DNS, Search Console, LinkedIn Post Inspector) are a manual checklist because they happen in external dashboards.

**Tech Stack:** Astro 7 static build, `astro-seo` (declarative meta-tag component), `@astrojs/sitemap` (already configured with `site: 'https://infrahats.com'`), headless Chrome for OG-image rendering, Lighthouse for the final audit.

**Current state (audited 2026-07-07):** ✅ canonical, meta description, robots.txt + sitemap, JSON-LD `ProfessionalService`, one `h1` + semantic `h2`s, self-hosted fonts. ❌ `og-image.png` referenced but missing (404 on share), title 72 chars / description ~230 chars (both truncated in SERPs), no `twitter:image` / `og:site_name` / `theme-color` / touch icon, JSON-LD lacks `logo`/`sameAs`/pricing, site not yet registered with any search engine.

**All commits in this plan: stage only the files named in the task** (user rule: partial staging, never sweep unrelated working-tree changes).

## Phase 2 (deliberately out of scope — revisit after launch)

A single-page site caps how many queries you can rank for. When there's appetite:
- **LinkedIn content engine (highest-leverage for this business)**: 1–2 posts/week from founders' personal profiles (war stories, cost-optimization wins, before/after architecture diagrams), each linking back to the site. LinkedIn is where DevOps consulting deals actually originate; treat the company page as the archive and personal profiles as the distribution.
- **Per-service pages** (`/services/kubernetes-consulting`, `/services/cloud-migration`, …) targeting one keyword each — Astro makes these cheap, and each gets its own title/description/JSON-LD `Service` **and its own og-image**, so every LinkedIn share of a service page gets a tailored card (reuse the Task 3 template with a different headline).
- **Blog posts**: the long-form home for the LinkedIn material — the realistic long-tail + backlink engine for a consultancy.
- **Case studies with named outcomes** once client permission exists — strongest conversion + E-E-A-T signal, and prime LinkedIn post material.

Not planned now (YAGNI): FAQ schema (no FAQ section on the page — schema without visible content violates Google guidelines), hreflang (English only), AMP (dead), paid tooling.
